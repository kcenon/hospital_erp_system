## 1. 프로젝트 개요

### 1.1 프로젝트명

**입원환자 관리 ERP 시스템 (Inpatient Management ERP System)**

### 1.2 프로젝트 배경

현재 구글 시트로 관리 중인 입원환자 데이터를 체계적인 시스템으로 전환하여:

- **데이터 보안 강화**: 환자 개인정보의 체계적 관리 및 접근 통제
- **업무 효율 향상**: 중복 입력 방지 및 실시간 정보 공유
- **기존 시스템 연동**: 진료 환자 관리 프로그램과의 데이터 통합

### 1.3 프로젝트 목표

```
┌─────────────────────────────────────────────────────────────────┐
│                        핵심 목표                                  │
├─────────────────────────────────────────────────────────────────┤
│  1. 기존 진료 프로그램과 실시간 데이터 연동                          │
│  2. 병실 현황 및 환자 상태 실시간 모니터링                           │
│  3. 회진/라운딩 보고의 모바일 지원                                  │
│  4. 의료법 및 개인정보보호법 준수                                   │
└─────────────────────────────────────────────────────────────────┘
```

## 2. 시스템 범위

### 2.1 플랫폼 구성

| 플랫폼     | 용도           | 사용자         |
| ---------- | -------------- | -------------- |
| PC Web     | 관리/원무 업무 | 원무과, 관리자 |
| Tablet Web | 회진/라운딩    | 의사, 수간호사 |
| Mobile Web | 간편 조회/입력 | 간호사, 의료진 |

### 2.2 기능 범위

```
입원환자 관리 ERP
├── 시스템 연동 (Interface)
│   ├── 기존 진료 프로그램 연동 (DB/API)
│   └── 환자 기초 정보 동기화
│
├── 병실 및 환자 관리
│   ├── 병실 현황판 (층별/호실별)
│   ├── 입원 수속 관리
│   ├── 전실 처리
│   └── 퇴원 처리 및 이력
│
├── 보고서 및 일지
│   ├── 입원환자 일일 보고서
│   │   ├── 바이탈 사인
│   │   ├── I/O (섭취/배설량)
│   │   ├── 식사 기록
│   │   └── 투약 기록
│   ├── 치료 보고서
│   └── 일일 라운딩 보고
│
└── 관리자 기능
    ├── 사용자 계정 관리
    ├── 권한 관리 (역할 기반)
    ├── 데이터 백업
    └── 감사 로그
```

## 3. 이해관계자

### 3.1 사용자 그룹

| 역할              | 주요 기능                              | 접근 권한          |
| ----------------- | -------------------------------------- | ------------------ |
| **시스템 관리자** | 시스템 설정, 사용자 관리, 감사 로그    | 전체               |
| **원무과**        | 입퇴원 수속, 병실 배정, 환자 정보 관리 | 환자관리, 병실관리 |
| **의사**          | 회진, 처방, 치료 보고서 작성           | 진료, 보고서       |
| **수간호사**      | 병동 관리, 간호 일지 검토, 라운딩 보고 | 병동 전체          |
| **간호사**        | 바이탈 입력, 간호 일지 작성, I/O 기록  | 담당 환자          |

### 3.2 외부 시스템

| 시스템             | 연동 방식             | 데이터                  |
| ------------------ | --------------------- | ----------------------- |
| 기존 진료 프로그램 | DB 직접 연결 또는 API | 환자 기초정보, 진료내역 |
| 구글 시트 (레거시) | 일회성 마이그레이션   | 기존 입원 데이터        |

## 4. 비기능적 요구사항

### 4.1 보안 요구사항

| 항목          | 요구사항                             |
| ------------- | ------------------------------------ |
| 데이터 암호화 | AES-256 (저장 시), TLS 1.3 (전송 시) |
| 인증          | 다중 인증(MFA) 지원                  |
| 접근 통제     | RBAC (역할 기반 접근 제어)           |
| 감사 로그     | 모든 데이터 접근/수정 이력 기록      |
| 세션 관리     | 자동 로그아웃, 동시 세션 제한        |

### 4.2 성능 요구사항

| 항목        | 목표       |
| ----------- | ---------- |
| 페이지 로딩 | 3초 이내   |
| API 응답    | 500ms 이내 |
| 동시 접속자 | 100명 이상 |
| 가용성      | 99.5% 이상 |

### 4.3 규정 준수

- **개인정보보호법**: 민감정보 암호화, 접근 기록 2년 보관
- **의료법**: 의무기록 5년 보관, 진료기록 열람 통제
- **전자문서법**: 전자서명, 무결성 보장

## 5. 제약사항 및 가정

### 5.1 제약사항

1. **기존 시스템 의존성**: 진료 프로그램 DB 구조에 따라 연동 범위 결정
2. **클라우드 환경**: 클라이언트 측 클라우드 계정 및 비용 부담
3. **네트워크 환경**: 병원 내부 네트워크 정책 준수 필요

### 5.2 가정사항

1. 기존 진료 프로그램의 DB 접근 권한 또는 API 제공 가능
2. 구글 시트 데이터의 정제 및 매핑 협조 가능
3. 사용자 교육 및 변화 관리 지원 가능
