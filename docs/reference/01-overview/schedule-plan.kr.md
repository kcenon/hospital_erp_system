## 2. 단계별 상세 일정

### 2.1 Phase 1: 분석 & 설계 (4주)

```
Week 1-2: 요구사항 분석
├── 현황 파악
│   ├── 기존 구글 시트 분석
│   ├── 현재 업무 프로세스 인터뷰
│   ├── 기존 진료 프로그램 DB 분석
│   └── 이해관계자 요구사항 수집

└── 요구사항 정의
    ├── 기능 요구사항 명세
    ├── 비기능 요구사항 명세
    ├── 우선순위 정의 (MoSCoW)
    └── 요구사항 검토 회의

Week 3-4: 시스템 설계
├── 아키텍처 설계
│   ├── 시스템 아키텍처 문서
│   ├── 데이터베이스 설계
│   ├── API 설계
│   └── 보안 설계
│
├── UI/UX 설계
│   ├── 와이어프레임
│   ├── 화면 설계서
│   └── 디자인 시스템 정의
│
└── 클릭 가능 프로토타입
    ├── 주요 화면 프로토타입
    ├── 사용자 피드백 수집
    └── 최종 설계 확정
```

## 8. 변경 관리

### 8.1 변경 요청 프로세스

```
┌─────────────────────────────────────────────────────────────────┐
│                        변경 요청 프로세스                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  1. 변경 요청 ──> 2. 영향 분석 ──> 3. 검토/승인 ──> 4. 구현     │
│       │               │               │               │         │
│       ▼               ▼               ▼               ▼         │
│   변경요청서       영향분석서       승인/반려       개발/테스트   │
│   작성            (일정, 비용)      결정            수행         │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 8.2 변경 승인 기준

| 변경 규모          | 승인 권한         | 처리 기한 |
| ------------------ | ----------------- | --------- |
| 경미 (UI 수정 등)  | PM                | 3일       |
| 보통 (기능 수정)   | PM + 고객 담당자  | 1주       |
| 대규모 (일정 영향) | 고객 의사결정권자 | 2주       |
